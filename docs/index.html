Entity Vault

Privacy-preserving entity resolution and controlled data sharing, built entirely on Snowflake.

Snowflake Native Open Architecture

Overview

What is the Entity Vault?

Entity Vault is a Snowflake-native platform for resolving, encoding, and sharing entity data across organizational boundaries without exposing raw identifiers. It provides a canonical identity layer where any entity type—people, companies, products, locations—can be resolved to a golden record, encoded with partner-specific irreversible keys, and shared under fine-grained policy controls.

Business Problem

Fragmented Entity Data

Organizations maintain disconnected representations of the same entities across systems. Customer "John Smith" exists as 47 different records with no canonical truth.

Cross-Partner Correlation Risk

Sharing a common identifier with multiple partners enables unauthorized cross-correlation. If Partner A and Partner B both receive the same key, they can merge datasets without consent.

Lack of Controlled Sharing

No mechanism exists to share enriched entity metadata while enforcing consent, suppression, and clearance policies at the field level.

Key Value Propositions

  • Privacy-Preserving: All external-facing identifiers are one-way HMAC hashes. Raw PII never leaves the vault.
  • Partner-Specific Encoding: Each partner receives a unique, deterministic key derived from a namespace-scoped HMAC. Cross-partner correlation is architecturally impossible.
  • Policy-Gated Access: Consent records, suppression lists, and clearance levels govern every data flow. No data moves without explicit policy authorization.

Industry Context

Advertising Identity Resolution Services

Commercial identity resolution platforms in the advertising and marketing ecosystem operate as middleware that maps between partner-specific identifier domains. Key patterns include:

  • Persistent Identifiers: A deterministic, people-based identifier that links across devices, cookies, and CRM systems
  • Transcoding: Translation between partner key spaces (e.g., Partner A's encoded ID → Partner B's encoded ID) without exposing the underlying identity
  • Cloud-Native Integration: Running as native applications within cloud data platforms, enabling identity resolution without data leaving the customer's environment
  • Data Marketplace: Operating marketplaces where data providers can monetize audiences via encoded segments

Key Insight: These services are fundamentally middleware companies that connect disparate identity systems. Their value lies in the network effect—the more partners connected, the more valuable transcoding becomes.

Healthcare Entity Matching Services

Healthcare-focused matching platforms provide tokenization that enables cross-dataset patient matching without sharing PII. Their approach is specifically designed for the regulatory constraints of HIPAA-governed healthcare data.

  • Tokenization: Replaces PII fields with encrypted tokens that are deterministic within a dataset but cannot be reversed to recover the original value
  • Matching Strategy: Uses ternary agreement vectors combined with machine learning models to determine match confidence across tokenized datasets
  • Privacy Architecture: Evaluates approaches like Bloom filters for matching but often chooses tokenization for superior privacy guarantees and audit trail capabilities
  • Healthcare Focus: Purpose-built for patient matching across claims, EHR, lab, and pharmacy datasets

Architecture

Snowflake-Native Schema Design

Entity Vault is organized into six logical schemas, each handling a distinct concern:

Schema Purpose Key Objects
CORE Canonical entity storage ENTITIES, IDENTIFIERS, METADATA
RESOLUTION Identity matching and deduplication RESOLUTION_LOG, MATCH_CANDIDATES
ENCODING Partner-specific key generation ENCODINGS, NAMESPACES, HMAC_KEYS
ENRICHMENT Contributed metadata and groups CONTRIBUTIONS, SCHEMAS, GROUPS
POLICY Access control and consent CONSENT_RECORDS, SUPPRESSIONS, POLICIES
AUDIT Complete operation history AUDIT_LOG, ACCESS_EVENTS

Resolution Strategy Chain

When an entity is submitted for resolution, the system evaluates match candidates through a tiered strategy:

1

Tier 1 — Exact Hash Match

SHA-256 hash of normalized identifiers is compared against the identifier index. Deterministic, zero ambiguity. If a match is found, the existing entity is returned immediately.

2

Tier 2 — Cortex Search Hybrid Match

Metadata fields are embedded and searched via Snowflake Cortex Search Service. Combines semantic similarity with structured attribute matching. Returns candidates above a configurable confidence threshold.

3

Tier 4 — Create New Entity

If no match exceeds the confidence threshold, a new canonical entity is minted with a system-generated UUID. The submitted identifiers and metadata are attached as the founding record.

HMAC-Based Encoding

Encodings are generated using HMAC-SHA256 with namespace-scoped secret keys. This ensures:

  • Irreversibility: An encoded key cannot be reversed to recover the entity UUID or any PII
  • Determinism: The same entity + namespace always produces the same encoded key, enabling join operations
  • Namespace Isolation: Partner A's key for Entity X is cryptographically unrelated to Partner B's key for the same entity
encoded_key = HMAC-SHA256(
    key   = namespace_secret,
    data  = entity_uuid || namespace_id
)
-- Output: 64-char hex string, unique per (entity, namespace) pair

Policy Engine

Every data flow passes through the policy engine, which evaluates three policy dimensions:

Consent Policies

Entity-level opt-in/opt-out records that control whether data can flow to a given namespace or use case.

Suppression Policies

Hard blocks that override all other policies. Used for regulatory holds, legal requests, and entity-initiated deletion.

Transcode Policies

Govern whether a key from Namespace A can be translated to Namespace B. Requires explicit bidirectional approval.

Clearance Levels provide a hierarchical access model for metadata fields:

  • LEVEL_1 — Public aggregate data (segment membership counts)
  • LEVEL_2 — Non-identifying enrichment (behavioral scores, propensity models)
  • LEVEL_3 — Identifiable attributes (name, email hash, phone hash)
  • LEVEL_4 — Full internal access (raw PII, system metadata)

Use Cases

1. Golden Record

Retrieve the canonical representation of an entity with all classified metadata. The golden record consolidates identifiers, enrichment data, and system metadata into a single authoritative view, filtered by the requester's clearance level.

GET /entity/{entity_id}/golden-record?clearance=LEVEL_2

Response:
{
  "entity_id": "uuid-...",
  "entity_type": "PERSON",
  "metadata": { ... filtered by clearance ... },
  "identifiers": [ ... non-PII identifiers ... ],
  "last_resolved": "2024-12-01T..."
}

2. Resolution

Submit one or more identifiers and/or metadata attributes to find the matching canonical entity. If no match exists above the confidence threshold, a new entity is created. The resolution tier used is returned in the response for transparency.

POST /resolve
{
  "identifiers": [{"type": "EMAIL_HASH", "value": "a1b2c3..."}],
  "metadata": {"first_name": "John", "zip": "94105"},
  "create_if_missing": true
}

3. Encoding

Generate a partner-specific encoded key for an entity. The key is deterministic (same inputs always yield same output) but irreversible. Partners use these keys as join keys in their own systems.

POST /encode
{
  "entity_id": "uuid-...",
  "namespace": "partner_a_prod"
}
// Returns: { "encoded_key": "7f3a9b..." }

4. Transcoding

Translate an encoded key from one partner namespace to another, subject to transcode policy approval. Both the source and destination namespaces must have an active transcode policy permitting the translation. The vault resolves the source key to the internal entity, then re-encodes for the target namespace.

POST /transcode
{
  "source_namespace": "partner_a_prod",
  "source_key": "7f3a9b...",
  "target_namespace": "partner_b_prod"
}
// Policy check: TRANSCODE_POLICY(A→B) must be ACTIVE

5. Enrichment Out

Batch retrieval of metadata for a set of entities, filtered by the requester's clearance level and active consent policies. Useful for building audience segments or enriching partner datasets with permissioned attributes.

POST /enrich/batch
{
  "entity_ids": ["uuid-1", "uuid-2", ...],
  "fields": ["age_range", "income_bracket", "interest_scores"],
  "namespace": "partner_a_prod",
  "clearance": "LEVEL_2"
}

6. Create Group

Define structured entity groupings—households, organizations, audience segments, or custom taxonomies. Groups are first-class objects with their own metadata, membership rules, and policy controls. Supports the group marketplace where partners can discover and subscribe to shared segment definitions.

POST /groups
{
  "group_type": "HOUSEHOLD",
  "name": "Smith Household - 94105",
  "members": ["uuid-1", "uuid-2", "uuid-3"],
  "metadata": {"address_hash": "d4e5f6..."}
}

7. Contributions (Enrich In)

Partners contribute metadata back to entities they have a relationship with. Contributions follow a schema-registered, versioned format. Each contribution is stamped with provenance (who, when, schema version) and becomes available to other authorized consumers based on policy.

POST /contribute
{
  "entity_id": "uuid-...",
  "namespace": "partner_a_prod",
  "schema": "purchase_behavior_v2",
  "payload": {
    "last_purchase_date": "2024-11-15",
    "lifetime_value_tier": "HIGH",
    "category_affinities": ["electronics", "outdoor"]
  }
}

Data Model

Key Tables & Relationships

IDENTIFIERS CORE Typed identifier values linked to entities (email hash, phone hash, device ID, etc.) FK → ENTITIES
METADATA CORE Key-value metadata with classification level and provenance tracking FK → ENTITIES, classified by level

NAMESPACES ENCODING Partner namespace definitions with HMAC key references Parent of ENCODINGS
CONTRIBUTIONS ENRICHMENT Versioned partner-contributed metadata payloads FK → ENTITIES, FK → SCHEMAS
GROUPS ENRICHMENT Named entity groupings with type and membership M:N with ENTITIES via GROUP_MEMBERS
CONSENT_RECORDS POLICY Per-entity, per-namespace consent status with effective dates FK → ENTITIES, FK → NAMESPACES
AUDIT_LOG AUDIT Immutable log of all vault operations with actor, action, and payload hash References all schemas

Metadata Classification

All metadata stored in the vault is classified into one of three tiers that determine visibility and policy enforcement:

Classification Visibility Examples Policy
INTERNAL System only Raw PII, source system IDs, internal scores Never exposed externally. Used for resolution only.
IDENTIFIABLE Clearance Level 3+ Hashed email, hashed phone, device IDs Requires explicit consent. Audit-logged on every access.
ENRICHMENT Clearance Level 2+ Age range, interest scores, segment membership Available to authorized partners per consent policy.

Classification Hierarchy: INTERNAL > IDENTIFIABLE > ENRICHMENT. A higher-classified field is never returned to a requester with insufficient clearance, regardless of other policy settings.

API & UI

FastAPI Backend

The backend is a FastAPI application that acts as a proxy layer between the React frontend and Snowflake. It handles authentication, request validation, and translates REST calls into Snowflake SQL/procedure invocations.

  • Architecture: Stateless proxy — no entity data is cached or stored in the application layer
  • Authentication: Snowflake connection credentials, with support for key-pair and OAuth flows
  • Endpoints: RESTful resources for each use case (resolve, encode, transcode, enrich, contribute, groups)
  • Validation: Pydantic models enforce request schema before Snowflake interaction
  • Error Handling: Snowflake errors are translated to appropriate HTTP status codes with structured error bodies
# Core endpoint structure
app = FastAPI(title="Entity Vault API")

@app.post("/resolve")       # Resolution with tiered strategy
@app.post("/encode")        # Namespace-scoped HMAC encoding
@app.post("/transcode")     # Policy-gated key translation
@app.post("/enrich/batch")  # Clearance-filtered metadata retrieval
@app.post("/contribute")    # Schema-validated metadata ingestion
@app.post("/groups")        # Entity group management
@app.get("/entity/{id}")    # Golden record retrieval

React Frontend

The frontend is a React single-page application with a dark-themed interface designed for data operations. It provides visual workflows for each use case and real-time feedback on resolution confidence, policy evaluation, and audit trails.

  • Theme: Dark interface matching the Snowflake design language (navy backgrounds, cyan accents)
  • Components: Entity search, resolution wizard, encoding dashboard, policy manager, contribution inbox
  • State Management: React hooks with context for session state (active namespace, clearance level)
  • Visualization: Resolution confidence scores, entity relationship graphs, group membership views

Demo Mode

The application supports a fully functional demo mode that generates random but realistic entity data on-the-fly. This enables exploration of all vault capabilities without requiring a populated Snowflake backend.

  • Random Generation: Synthetic entities with plausible names, identifiers, and metadata distributions
  • Simulated Resolution: Demonstrates tier progression with configurable match rates
  • Policy Simulation: Pre-configured consent scenarios showing permit/deny flows
  • No Dependencies: Runs entirely client-side when the backend is unavailable

Future Directions

Ontology Layer

A formal OWL-style metadata system that enables semantic queries across entity types, contribution schemas, and group structures. The ontology layer introduces 22 ONT_* tables that define classes, properties, and relationships in a machine-readable format.

ONT_CLASS

Defines entity type hierarchies (PERSON is-a ENTITY, EMPLOYEE is-a PERSON). Enables inheritance of properties and policies.

ONT_PROPERTY

Typed metadata field definitions with domain/range constraints. Maps contribution schema fields to semantic meaning.

ONT_RELATIONSHIP

Named, typed edges between entity classes (PERSON works_at COMPANY, PRODUCT sold_at LOCATION). Enables graph traversal queries.

Integration with the ontology-stack-builder enables automated generation of all 22 ONT_* tables from existing schema introspection, bootstrapping the ontology from the data that already exists in the vault.

Snowflake Postgres Backend

A hybrid write/read architecture that uses Snowflake Postgres for the transactional write path and Snowflake's analytical engine for the read path.

  • Write Path: Entities, identifiers, and encodings are written via Postgres, providing ACID transactional consistency and sub-millisecond write latency
  • CDC Mirroring: Change Data Capture streams Postgres writes to Snowflake tables in near-real-time, maintaining a synchronized read replica
  • Read Path: Cortex Search Service indexes the mirrored Snowflake data, enabling the hybrid resolution strategy at analytical scale
  • Benefits: Transactional consistency for writes (critical for encoding determinism), Snowflake's elastic compute for reads and AI-powered resolution

Contribution Marketplace Scaling

Evolving the contribution system into a full marketplace with quality guarantees and economic incentives:

  • Versioned Schemas: Formal schema evolution with backward/forward compatibility rules, deprecation timelines, and migration paths
  • Quality Scoring: Automated assessment of contribution freshness, coverage, accuracy, and consistency against the canonical record
  • Freshness Tracking: SLA-driven monitoring that alerts when contributions exceed their declared refresh interval
  • Economic Model: Contribution value scoring based on uniqueness, coverage gap filling, and downstream consumption metrics

Multi-Entity Resolution

Extending the resolution engine to handle cross-type entity relationships, enabling graph-style queries across the entity universe:

  • Cross-Type Links: PRODUCT belongs_to COMPANY, sold_at LOCATION, purchased_by PERSON
  • Relationship Resolution: Given a PERSON entity, resolve all related COMPANY entities via employment records
  • Graph Traversal: Multi-hop queries (find all PRODUCTs sold at LOCATIONs owned by COMPANY X)
  • Federated Resolution: Resolve entities using relationship context as additional signal (person at this company at this address)
